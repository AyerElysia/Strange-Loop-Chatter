# 意识模拟插件路线图整理版

## 现状

现在这套系统已经不是“概念设计”，而是已经有了可用的核心骨架。

已经比较明确、值得继续围绕它们展开的模块是：

- `drive_core_plugin`
- `self_narrative_plugin`
- `unfinished_thought_plugin`
- `diary_plugin`
- `booku_memory`
- `memory_passive_trigger`
- `proactive_message_plugin`
- `thinking_plugin`
- `prompt_logger`

这意味着当前最该做的，不是继续加很多概念性新插件，而是把这些现有模块整理成一条清晰的内部状态链。

## 为什么要整合

这里说的“整合”，不是把所有代码硬塞进一个大插件里，而是把它们放进同一个清晰的内部模型里。

之所以要这么做，主要有四个原因：

1. **减少职责重叠**
   - 现在很多模块都在描述“内部状态”，但如果彼此没有统一边界，就容易重复、打架、互相污染。

2. **让状态之间真的能联动**
   - 自我叙事、未完成念头、记忆、主动性这些东西如果分得太散，就很难形成一个连续的内部过程。

3. **让主模型看到的是投影，不是原始数据库**
   - 目标不是把所有记录塞给主回复模型，而是让它看到“当前该怎么表现”的压缩结果。

4. **方便后续扩展**
   - 先整成清晰的层级，后面你想加情绪、关系、注意力、欲望时，就知道该放在哪一层，而不是再新开一套互不相干的系统。

所以这次整合的核心目标是：

- 让状态层有边界
- 让工作记忆能回流
- 让自我模型能持续更新
- 让 `drive_core` 真正做编排

## 整合后还能做什么

整合之后，依然应该保留两件事：

1. **能够自己找问题，然后分步骤解决**
   - 这件事应该由 `drive_core_plugin` 负责。
   - 它先从内部状态里挑出当前最值得追的主题，再把问题拆成步骤，逐步推进。
   - 也就是说，整合不是取消“思考”，而是让“思考”有统一入口。

2. **不影响主动消息**
   - 主动消息应该继续保留，但它不应该是独立乱跑的定时器。
   - 它应该受到关系温度、注意力预算、疲惫程度、未完成念头和当前课题的共同约束。
   - 这样主动消息才是“状态驱动的主动”，不是“时间到了就发”。

换句话说：

- `drive_core` 负责“现在我该想什么”
- `proactive_message_plugin` 负责“现在我该不该主动”
- `memory_passive_trigger` 负责“什么东西会突然浮上来”
- `self_narrative_plugin` 负责“我怎么看自己”
- `unfinished_thought_plugin` 负责“我还挂着什么”

这些功能不是互相替代，而是互相喂输入。

## 怎么统合才不影响原有功能

这是最关键的问题。

我认为正确的整合方式，不是把所有 Agent 合并成一个巨无霸，而是建立一个**统一的内部状态底座 + 统一的编排入口 + 各自独立的能力模块**。

### 1. 统一状态底座

所有 Agent 都写入同一个“内部状态黑板”或者状态层，但写的是不同种类的内容。

比如：

- `diary_plugin` 写经历
- `booku_memory` 写可检索记忆
- `unfinished_thought_plugin` 写未完成张力
- `self_narrative_plugin` 写自我理解
- `memory_passive_trigger` 写“可能被唤起的记忆信号”

关键点是：

- 这些模块可以共享底层状态结构
- 但它们不需要直接互相调用
- 它们只通过统一状态层读写

### 2. 统一编排入口

所有“要不要做下一步”的判断，都尽量收口到 `drive_core_plugin`。

`drive_core` 不直接替代别的模块，而是做总调度：

- 读当前状态
- 生成当前问题
- 决定优先级
- 决定交给谁执行
- 收集执行结果
- 更新状态

这样每个模块仍然保留自己的职责，但不会各自做各自的总决策。

### 3. 各 Agent 只做自己的事，不直接聊天

你提到“各 Agent 不能相互交流”，我认为这个判断是对的。

最稳的方案不是让 Agent 彼此对话，而是：

- 它们都写黑板
- `drive_core` 统一读黑板
- `drive_core` 决定谁先做什么
- 需要协同的时候，通过状态传递，不通过 Agent 直接私聊

这样可以避免：

- 状态污染
- 循环解释
- 一个 Agent 影响另一个 Agent 的措辞风格
- 任务边界模糊

### 4. 主动消息和内部思考不要抢同一条控制权

主动消息、内在思考、记忆浮现，这三者不是同一件事。

建议分工如下：

- `proactive_message_plugin` 负责外显行为
- `drive_core_plugin` 负责是否触发、触发什么、触发到什么程度
- `memory_passive_trigger` 负责记忆被什么东西唤起
- `unfinished_thought_plugin` 负责哪些东西还没结束

这样即使功能很多，也不会抢同一个按钮。

## 现在的结构

我建议把现有体系直接收敛成四层。

### 1. 状态层

这一层负责描述“当前内部是什么样子”，但不直接决定回复内容。

适合放进这一层的内容：

- 情绪倾向
- 注意力余量
- 关系温度
- 当前疲惫程度
- 依恋 / 距离倾向

这层不一定需要马上拆成很多独立插件，先可以作为 `drive_core_plugin` 的内部状态维度。

### 2. 工作记忆层

这一层负责保存“现在还挂着什么”。

适合放进这一层的内容：

- 未完成念头
- 最近记忆
- 当前课题
- 正在推进的自我发问

这一层的作用是让角色能持续想着同一件事，而不是每轮都重新起步。

### 3. 自我模型层

这一层负责“我如何理解自己”。

适合放进这一层的内容：

- 自我叙事
- 稳定边界
- 反复模式
- 长期张力

这一层不负责具体行动，而负责让角色在长对话里保持身份连续性。

### 4. 编排控制层

这一层应该由 `drive_core_plugin` 承担。

它负责：

- 读取上面三层状态
- 判断当前是否该主动
- 选择该聊什么
- 决定是展开、收缩、追问、回避，还是转题
- 把压缩后的状态投影注入主回复

换句话说，`drive_core` 不只是“保存工作区”，而应该是整个内部系统的调度器。

## 已经落地的部分

### 核心已经成形

- `drive_core_plugin` 已经能承担上层编排
- `self_narrative_plugin` 已经能维护自我理解
- `unfinished_thought_plugin` 已经能维护未完成张力
- `diary_plugin` 和 `booku_memory` 已经能提供记忆沉淀
- `memory_passive_trigger` 已经能帮助记忆浮现
- `proactive_message_plugin` 已经能承担主动消息行为
- `thinking_plugin` 已经提供了思考动作相关能力

### 调试和观测已经有基础

- `prompt_logger` 可以记录完整提示词
- 这对判断各层状态有没有真正进入主模型非常关键

## 当前不纳入这套体系的东西

下面这两个插件，我建议先明确排除在当前意识模拟核心体系之外：

- `kokoro_flow_chatter`
- `sleep_wakeup_plugin`

它们可以继续作为独立能力存在，但不再作为这套路线图的组成部分。

这样能避免路线图变得太散，也能让重点更集中。

## 推荐优先级

### 第一优先级：把 `drive_core_plugin` 做成真正的总控

不是再加很多新模块，而是让 `drive_core` 真正负责组织内部状态读取和决策。

它应该直接消费：

- 自我叙事
- 未完成念头
- 连续记忆
- 关系温度
- 最近对话节奏

### 第二优先级：补关系温度

如果目标是让角色会主动找话题，关系温度比单纯情绪更关键。

因为它直接决定：

- 对谁主动
- 主动到什么程度
- 用什么语气主动
- 是靠近还是保持距离

### 第三优先级：补注意力预算

注意力预算决定角色是否愿意认真展开、是否容易跑题、是否更适合简答。

这个层级对“像一个会累、会分心的主体”很重要，但不需要做得太复杂。

### 第四优先级：把主动性接到状态上

主动消息不应该只是时间到了就发，而应该结合：

- 关系温度
- 疲惫程度
- 当前未完成念头
- 最近记忆是否有回流

这样角色主动找话题才不会像机械定时器。

## 不建议继续扩大的方案

### 1. 不要把所有状态都拆成独立插件

如果每个状态都独立成插件，最后会变成很多小岛，彼此不理解。

更稳的做法是：

- 先在 `drive_core` 里聚合
- 只把真正稳定、长期需要单独维护的层，再拆出去

### 2. 不要让 LLM 每轮重写人格

人格感来自连续变化，不来自每轮重新生成整个人设。

所以应该避免：

- 每轮都整段重写“我是谁”
- 每轮都把全部状态重新总结成一个大段 prompt

### 3. 不要把内部状态直接暴露成原始数据库

主回复模型看到的应该是状态投影，不是全量记录。

也就是：

- 给它看“当前想什么”
- 不要给它看“所有历史怎么变的”

### 4. 不要让各 Agent 互相直接通信

如果让各 Agent 直接对话，最后很容易变成：

- 彼此重复解释
- 彼此覆盖状态
- 彼此抢判断权

更稳的方式是：

- Agent 对黑板写结果
- `drive_core` 对黑板做归纳
- 主回复模型只看压缩后的投影

## 推荐的最小闭环

如果只做一个最小但有效的意识模拟闭环，建议是：

1. `self_narrative_plugin` 提供自我解释
2. `unfinished_thought_plugin` 提供未完成张力
3. `diary_plugin` 和 `booku_memory` 提供沉淀素材
4. `memory_passive_trigger` 负责让记忆适时浮现
5. `drive_core_plugin` 读取这些状态并生成当前课题
6. `drive_core_plugin` 决定要不要主动、要不要转题、要不要追问
7. 主回复模型只接收压缩后的状态投影

这个闭环跑通以后，角色就会明显更像一个持续存在的主体。

## 下一步执行顺序

我建议按下面顺序继续做：

1. 先把 `drive_core_plugin` 的职责边界定清楚
2. 再把 `self_narrative_plugin`、`unfinished_thought_plugin`、`diary_plugin`、`booku_memory` 的输入链稳定住
3. 再把 `proactive_message_plugin` 和 `memory_passive_trigger` 接到状态上
4. 最后再微调 `prompt_logger`，用来检查主模型到底看到了什么

## 结论

这份路线图现在最适合的形态，不是“再列很多未来插件”，而是：

- 保留真正已经成型的核心层
- 删掉不纳入本轮体系的旧路线
- 把 `drive_core_plugin` 变成总控
- 把主动性、记忆、自我模型串成闭环

如果按这个方向推进，后续扩展会更稳，也更容易让角色表现出真正的持续内部状态。
