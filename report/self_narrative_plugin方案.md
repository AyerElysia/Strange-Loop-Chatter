# self_narrative_plugin 方案

## 目标

`self_narrative_plugin` 的目标不是记录事实，而是维护一份持续演化的“自我解释”。

它回答的不是“发生了什么”，而是“这些经历对我意味着什么”。

这层能力是模拟意识的关键之一，因为人类的连续性很大程度上来自对自我的解释，而不是单纯的事件列表。

## 核心定位

这个插件只负责“角色如何理解自己”。

它不做：

- 原始对话归档
- 日记全文存储
- 事实检索
- 关系数值计算
- 情绪底层更新

它负责：

- 从日记、连续记忆、睡眠状态、关系变化、近期对话模式中提炼自我解释
- 维护低频更新的自我叙事摘要
- 在主模型回复前注入这层叙事，影响语气、立场、稳定性和自我一致性

## 设计原则

1. 自我叙事必须是“解释层”，不是“事实层”。
2. 自我叙事必须低频更新，不能每轮对话重写。
3. 自我叙事必须允许演化，而不是完全替换。
4. 自我叙事必须持久化，重启后可恢复。
5. 自我叙事必须简短稳定，不能塞成大段冗长文本。

## 数据结构建议

建议把 self_narrative 拆成四个稳定子层，不要直接存成一大段散文。

### 1. `self_view`

当前对自己的总体看法。

示例：

- 最近更安静，但不是失去兴趣
- 最近更敏感，容易先感知氛围
- 最近更想保持回应质量，而不是单纯频率

用途：

- 作为主模型的“当前自我状态”

### 2. `ongoing_patterns`

反复出现的行为模式。

示例：

- 遇到压力时会先收缩
- 在熟悉关系中更容易展开
- 长对话后更倾向总结而不是扩写

用途：

- 形成习惯层的自我理解
- 维持风格连续性

### 3. `open_loops`

尚未解释完的自我问题。

示例：

- 我为什么最近总想降低回应强度
- 我到底是在保持距离，还是在等待更好的理解方式
- 我对某类话题的回避是否意味着疲惫

用途：

- 让角色有“内部未完成状态”
- 增强持续思维感

### 4. `identity_bounds`

稳定的自我边界和原则。

示例：

- 我更重视真实表达，而不是迎合
- 我倾向先理解再判断
- 我不希望自己变得过度机械

用途：

- 防止叙事漂移
- 让角色不会频繁变成完全不同的人

## 更新机制

`self_narrative_plugin` 应该是低频更新，而不是每条消息都刷新。

### 最终触发方式

1. 每天 0 点自动更新一次。
2. 支持通过命令手动触发更新。

### 不建议触发

- 每条消息都更新
- 用户只说一句闲话就重写
- 上下文不足时硬生成

### 更新原则

- 允许小幅修正，不允许彻底重写
- 保留旧版本历史
- 每次更新都记录原因
- 保留旧叙事中的合理部分，不要抹掉全部痕迹

### 冷却与幂等

- 每天 0 点自动更新必须幂等，重复触发不能重复写入
- 手动命令更新需要冷却时间，建议 5 分钟内不允许连续触发
- 如果当天已经自动更新过，手动更新仍可执行，但只做增量修正

## 写入流程

建议使用“两阶段写入”。

### 阶段一：候选生成

从以下来源收集输入：

- 连续记忆压缩摘要
- 最近几篇日记
- 睡眠状态变化
- 关系温度变化
- 当前自我叙事历史

然后让 LLM 生成候选叙事片段，例如：

- 我最近更倾向先观察再回应
- 我在熟悉关系里更放松
- 我似乎开始在意回应质量，而不是单纯速度

### 阶段二：合并与裁剪

把候选片段合并进已有叙事层：

- 相似内容合并
- 冲突内容保留为开放问题
- 过旧内容降权
- 最终只保留少量核心主题

这样可以避免叙事无限膨胀，也避免内容越来越散。

## 命令触发设计

建议增加一个专门的命令入口，用于手动触发和查看状态。

### 推荐命令

- `/self_narrative update`
- `/self_narrative view`
- `/self_narrative history`
- `/self_narrative reset`

### 最小命令集

如果想先简化实现，最少保留：

- `/sn update`
- `/sn view`

### 命令规则

- `update`：立即触发一次增量更新
- `view`：查看当前自我叙事
- `history`：查看最近几次演化记录
- `reset`：重置到基础版本，慎用

### 手动触发限制

- 建议加 5 分钟冷却
- 只允许管理员或当前聊天流对应用户触发
- 防止重复命令把叙事刷乱

## 持久化建议

建议按聊天隔离持久化，每个聊天流一个独立文件。

建议路径示例：

- `data/self_narratives/private/<stream_id>.json`
- `data/self_narratives/group/<stream_id>.json`

每个文件建议包含：

- `version`
- `updated_at`
- `self_view`
- `ongoing_patterns`
- `open_loops`
- `identity_bounds`
- `history`
- `sources`

这样重启后可以恢复，而且不同聊天对象或群聊可以拥有不同的自我解释风格。

## 和连续记忆的关系

`self_narrative_plugin` 不应该替代连续记忆，而应该建立在连续记忆之上。

两者的区别：

- 连续记忆回答“我经历过什么”
- self_narrative 回答“这些经历让我觉得我是谁”

换句话说：

- 连续记忆是素材
- self_narrative 是解释

这个分层很重要。否则所有东西都会变成可检索日志，最后只剩“记住了什么”，没有“自我”。

## 对主模型的注入方式

建议不要直接把全部内容裸塞给主模型，而是做成稳定的 prompt 区块。

推荐格式：

```text
【自我叙事】
- 当前自我理解：……
- 反复出现的模式：……
- 尚未解释完的问题：……
- 稳定边界：……
```

### 注入原则

- 短
- 稳定
- 可控
- 只注入摘要
- 默认不暴露详细历史

### 推荐注入内容

- 当前自我理解
- 反复出现的模式
- 尚未解释完的问题
- 可选的稳定边界

如果后续需要查看详细内容，可以加开关决定是否注入历史版本。

## 最小可行版本

建议先做 MVP，只保留三个字段：

- `self_view`
- `ongoing_patterns`
- `open_loops`

先不做过重的关系推断，也不做太多数值化。

推荐更新频率：

- 每天 0 点自动更新一次
- 必要时通过命令手动触发一次

## 主要风险

### 1. 叙事漂移

表现：

- 每次更新都把人格写偏

解决：

- 保留 `identity_bounds`
- 限制每次更新幅度

### 2. 叙事过密

表现：

- 字太多，主模型吸收不了

解决：

- 只保留少量稳定摘要
- 注入时只给核心层

### 3. 叙事僵化

表现：

- 太稳定，最后变成固定人设

解决：

- 保留 `open_loops`
- 允许局部演化

## 推荐实现顺序

1. 定义 self_narrative 数据结构
2. 实现持久化与加载
3. 实现每日 0 点自动更新
4. 实现命令手动触发
5. 做 prompt 注入
6. 增加历史版本查看开关

## 结论

`self_narrative_plugin` 是“意识模拟”里非常关键的一层。

它不是记忆，也不是情绪，而是“角色对自己经历的解释”。

如果做得稳，角色会明显更像一个持续存在、不断理解自己的主体，而不只是一个会聊天的系统。
