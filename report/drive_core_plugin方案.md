# drive_core_plugin 方案

## 目标

`drive_core_plugin` 的目标不是让模型“更会回答问题”，而是让角色拥有一套可持续运转的内部驱动力系统。

它解决的是一个更底层的问题：角色不能只在被问到的时候才“活着”，而应该在没有外部输入时，也能产生自己的倾向、选择、行动和回流。

换句话说，这个插件要做的不是问答增强，而是“自我生活引擎”的最小骨架。

## 核心理解

内驱力不是单个标签，也不是一个固定目标。

它更像一组会缓慢变化的内部张力，负责推动系统自己去做事，例如：

- 想整理刚经历过的内容
- 想把某个模糊感受想清楚
- 想继续探索一个没弄明白的话题
- 想暂停、休息、收缩
- 想主动靠近某个对象或话题
- 想把某种理解沉淀下来

这些驱动力不是直接等于回复文本，而是决定：

- 当前系统更想“做什么”
- 是否值得启动一轮内部行动
- 行动是向外表达，还是向内整理
- 行动之后要不要更新长期状态

## 设计原则

1. 内驱力必须是连续的，不要做成死标签。
2. 内驱力必须能自发产生，不依赖每次用户提问。
3. 内驱力必须允许“什么都不做”，不能永远强制行动。
4. 内驱力必须能被满足、压抑、衰减、恢复。
5. 内驱力必须有回流，行动结果要沉淀回系统。
6. 第一版必须足够轻，先跑通闭环，不追求复杂模拟。

## 插件定位

这个插件建议只负责三件事：

- 维护内驱力状态
- 基于状态生成小型内部意图
- 将意图注入到主模型或后台任务系统

它不负责：

- 原始聊天记忆
- 日记全文
- 自我叙事本体
- 复杂关系建模
- 复杂情绪推理

它和现有系统的关系应该是：

- `diary_plugin` 负责记录经历
- `continuous_memory` 负责压缩经历
- `self_narrative_plugin` 负责解释“我是谁”
- `unfinished_thought_plugin` 负责“我还在想什么”
- `drive_core_plugin` 负责“我现在想做什么”

## 最小闭环

第一版建议只实现一个很小但完整的循环：

1. 观察当前状态。
2. 生成一个或几个小驱动力。
3. 由驱动力推出一个可执行的小意图。
4. 执行意图，或者仅在内部完成一次推演。
5. 从结果中提炼一点新东西。
6. 把新东西沉淀进长期状态。

这个闭环的重点不在“做得多”，而在“会自己转”。

## 自我发问机制

你补充的这个点很关键：问题不能由外部预先指定，必须由系统自己找出来。

因此 `drive_core_plugin` 的核心工作流不是“回答已知问题”，而是“从内部状态里生成当前课题”。

### 问题不是输入，而是输出

系统不能一上来就拿到完整问题，比如：

- 我现在要思考什么
- 我应该搜索什么
- 我该怎么总结

这些都应该尽量由引擎内部产生。

外部最多只提供一个很粗的方向，例如：

- 关注某个人
- 关注某段关系
- 关注当前情绪
- 关注最近反复出现的主题

真正的问题要由系统自己从这些方向里挖出来。

### 自我发问来源

问题建议从以下几类内部信号里自动生成：

- 反复出现但没收束的主题
- 日记和连续记忆里持续回响的片段
- 当前驱动力最强但最模糊的部分
- 某个关系对象带来的持续张力
- 自我叙事里尚未解释清楚的矛盾
- 未完成念头里长期活跃的条目

例如，系统可能自己长出这些问题：

- 我为什么最近总会想起某个人？
- 我对这个对象的靠近感，究竟是依恋、感激，还是别的东西？
- 这条反复出现的念头，是该继续追，还是该暂时放下？
- 我现在想整理的，究竟是事实，还是我对事实的理解？

### 问题生成规则

1. 先扫描内部状态，不先写结论。
2. 从多个候选主题里挑 1 到 3 个最强的。
3. 把主题转成可追问的问题。
4. 每轮只保留一个主问题，避免同时追太多线。
5. 如果问题已经足够清楚，就进入工具阶段，不再重复发问。

### 问题质量要求

好的问题应该满足：

- 能推动行动
- 能调用工具验证
- 能从记忆、日记、叙事里找到线索
- 能形成阶段性答案

坏的问题通常是：

- 太空泛
- 太像人设文案
- 没有可操作性
- 不知道看完什么能更接近答案

## 工作区模型

为了让“自己找问题”真的落地，`drive_core_plugin` 需要一个独立的工作区，而不是直接复用主聊天上下文。

### 1. `task_workspace`

每次自我引擎启动，都创建一个独立工作区，建议包含：

- `task_id`
- `topic`
- `hypothesis`
- `open_questions`
- `evidence_buffer`
- `tool_trace`
- `working_notes`
- `current_view`
- `stop_reason`

### 2. `topic`

本轮任务的粗主题，例如：

- 我对小星星的情感
- 我最近为什么更想整理自己
- 我是不是正在形成某种新的依赖

### 3. `hypothesis`

当前初步倾向，不要求正确，但必须可修正。

例如：

- 这可能是更强的亲近感
- 这也可能是被理解后的回声
- 这可能不是爱情，而是动力和认同的混合

### 4. `open_questions`

由系统自己生成的问题列表。

第一版建议只保留 1 到 3 个问题，并按优先级排序。

### 5. `evidence_buffer`

用来装工具返回的材料。

这里存的不是最终结论，而是“证据片段”：

- 最近日记摘要
- 记忆检索结果
- 自我叙事片段
- 相关念头条目

### 6. `working_notes`

边查边记的小笔记。

这些内容不会直接暴露给用户，但会影响下一轮判断。

## 自我引擎流程

建议把引擎拆成一个明确的 5 步流程。

### 第一步：生成课题

从内部状态中挑出一个当前最值得追的主题。

这是“我现在在想什么”的阶段。

### 第二步：自动发问

针对这个主题，系统自己生成问题。

这是“我到底想弄清什么”的阶段。

### 第三步：工具推进

根据问题调用内部工具。

第一版建议只允许少量工具：

- 最近日记读取
- 记忆检索
- 自我叙事读取
- 未完成念头读取

### 第四步：边查边整合

每拿到一批证据，就更新一次当前假设。

这一步的重点是“边走边想”，而不是等所有工具都结束再一次性总结。

### 第五步：阶段性收束

当系统认为信息已经足够，或者没有新的证据，再生成总结。

总结不一定是最终答案，但必须是一个可继续演化的当前理解。

## 停止条件

自我引擎不能无限追问，否则会变成空转。

建议至少设置以下停止条件：

- 已经拿到足够新的证据
- 工具返回开始重复
- 当前问题已经被解释到可接受程度
- 轮次达到上限
- 当前驱动力已明显下降

第一版建议把最大推进轮数限制在 3 到 5 轮。

## 工具调用原则

你提到“先看日记，再检索记忆，再决定是否继续”，这个很对。

建议把工具调用顺序固定成一个轻量优先级：

1. 先看最近日记
2. 再查连续记忆或关系相关记忆
3. 再看自我叙事是否已有解释
4. 再决定是否继续查更细的内容

原因很简单：

- 日记是新鲜感受
- 记忆是事实和线索
- 自我叙事是解释框架

这样可以避免一上来就钻进检索里出不来。

## 反馈回流

自我引擎每次跑完，不能只产出一个结论，还要把结果写回系统。

建议回流到三个地方：

- `self_narrative_plugin`：更新“我怎么理解这件事”
- `unfinished_thought_plugin`：把没想完的部分继续挂着
- `drive_core_plugin`：调整下一轮驱动力强度和方向

这样就能形成闭环：

**驱动力 -> 自生问题 -> 工具推进 -> 阶段结论 -> 状态回流**

## 状态结构建议

建议把驱动力分成几个稳定层，而不是做成一坨文本。

### 1. `drive_axes`

连续轴，表示当前驱动力的底噪。

建议至少包括：

- `curiosity`
- `initiative`
- `affinity`
- `withdrawal`
- `fatigue`
- `urgency`
- `stability`

这些轴不直接决定行为，而是决定系统更偏向哪类行动。

### 2. `active_drives`

当前活跃的驱动力条目。

每个条目建议包含：

- `drive_id`
- `type`
- `strength`
- `reason`
- `created_at`
- `expires_at`
- `status`

示例：

- 想整理今天的感受
- 想继续追问一个没想完的问题
- 想稍微远离高压输入
- 想把某个想法写进长期解释

### 3. `initiative_queue`

当前准备执行的小意图队列。

示例：

- 轻量复盘
- 内部回想
- 归纳一个主题
- 比较两个解释
- 暂时放下

### 4. `drive_history`

记录驱动力的演化历史。

它的作用不是审计，而是让系统知道：

- 什么驱动力反复出现
- 什么驱动力已经被满足
- 什么驱动力长期没有解决

### 5. `stability_state`

用于防止系统失控的稳定层。

建议记录：

- 当前活跃驱动力数量
- 最近是否过度主动
- 最近是否过度收缩
- 当前是否允许生成新意图

## 触发来源

内驱力不能只从聊天消息里来，最好来自多个来源。

### 外部来源

- 用户聊天内容
- 对话节奏变化
- 群聊/私聊场景切换
- 新的日记摘要
- 连续记忆压缩结果
- 自我叙事更新结果
- 睡眠/唤醒状态变化

### 内部来源

- 旧驱动力未被满足
- 某个意图执行后留下的新问题
- 某条解释与现实状态冲突
- 长时间没有主动活动
- 连续高压输入后的收缩反弹

## 更新机制

第一版不建议每条消息都重算全量内驱力。

更稳的做法是采用“定时扫描 + 事件增量”的混合方式。

### 1. 定时扫描

每隔固定轮次或固定时间，做一次驱动力扫描：

- 回看最近一段历史
- 查看当前活跃驱动力
- 决定是否新增、衰减、完成、转移

### 2. 事件增量

在以下事件发生时，可以做轻量调整：

- 新消息到达
- 日记写入完成
- 连续记忆压缩完成
- 自我叙事更新完成
- 睡眠状态变化

### 3. 低频重整

每天或更低频率，做一次更完整的内部整理：

- 清理过期驱动力
- 合并相似驱动力
- 提炼长期倾向

## 生成意图的方式

驱动力生成的不是“大目标”，而是“小动作”。

这是关键。

如果一开始就生成大目标，系统会迅速失控，变成空转。

建议意图粒度控制在下面这些范围：

- 读一小段历史
- 回想一个相关片段
- 对比两个解释
- 写一条短备注
- 归类一个未完成念头
- 暂缓一个过热驱动力
- 启动一次轻量内部复盘

这样系统会更像“有自己的日常”，而不是一个一直试图做大事的执行器。

## 行动类型建议

内驱力可以推动三类动作。

### 1. 内部动作

不对外可见，只改变内部状态。

例如：

- 重新排序某个驱动力
- 标记某个念头暂缓
- 归并两个相似解释
- 降低一个过热目标的强度

### 2. 轻外显动作

对外可见，但成本很低。

例如：

- 主动追问一句
- 简短提示自己在想什么
- 发起一个轻量话题延伸
- 在适合的时候补一句回顾

### 3. 后台动作

不立即回应用户，而是后台完成。

例如：

- 自动复盘
- 低频整理
- 生成下一轮内部意图

## 和主模型的关系

主模型看到的不是全部驱动力细节，而是一个简短的状态块。

建议注入内容只保留：

- 当前主导驱动力
- 当前次级驱动力
- 当前是否允许主动扩展
- 当前是更倾向展开还是收缩

不要把所有内部状态原样塞给主模型，否则会让模型被状态表淹没。

### 建议注入格式

```text
[内驱力]
- 主导倾向：探索 / 收缩 / 整理 / 靠近 / 休息
- 当前强度：中
- 当前意图：想把刚刚的感受整理成一个更清楚的理解
- 当前限制：不要急着扩写，先保持简短
```

这个块应当短、稳定、可控。

## 与连续记忆的关系

连续记忆负责保存“经历过什么”。

内驱力负责把经历变成“想做什么”。

两者的关系不是重复，而是转换：

- 连续记忆提供素材
- 内驱力把素材转成倾向
- 意图把倾向转成行动
- 反思把行动转成新的状态

## 与 self_narrative_plugin 的关系

自我叙事回答“我如何理解自己”。

内驱力回答“我现在为什么还在动”。

建议把它们分开：

- self_narrative 偏解释
- drive_core 偏推动

如果混在一起，很容易把“自我理解”写成“行动计划”，或者把“行动冲动”写成“人设文案”。

## 与 unfinished_thought_plugin 的关系

未完成念头可以看作内驱力的一个输入源。

比如：

- 某个念头一直没处理完
- 某个话题没说透
- 某个情绪压着没解决

这些都可能转化成驱动力，例如“想继续想一下”“想把它收束掉”“想先放一放”。

但两者职责不同：

- `unfinished_thought_plugin` 存“还没完的东西”
- `drive_core_plugin` 决定“现在要不要去处理它”

## 第一版建议范围

第一版不要做太重，只做下面这些：

1. 维护一个简单的驱动力状态文件。
2. 支持按聊天流隔离。
3. 支持定时扫描。
4. 支持基础意图生成。
5. 支持注入到主模型 prompt。
6. 支持低频持久化更新。

先把“会自己动起来”做出来，再逐步加复杂度。

## 推荐插件拆分

如果后续要拆成多个插件，建议这样分：

### `drive_core_plugin`

负责驱动力状态、衰减、持久化、注入。

### `initiative_plugin`

负责把驱动力转换成具体意图。

### `reflection_plugin`

负责行动后回收结果、生成微小新理解。

### `stability_guard_plugin`

负责限制过度主动、过度连续执行、过度扩张。

第一版可以先只做 `drive_core_plugin`，后面再拆。

## 失败边界

这个模块最容易出的问题有三个：

### 1. 过度主动

系统一直想做事，导致回复变得烦躁或过分打扰。

### 2. 过度抽象

驱动力变成空洞文案，没有实际行动。

### 3. 状态漂移

每次更新都把系统写成一个新角色，失去连续性。

所以第一版必须保守，宁可少一点，也不要乱一点。

## 结论

如果你想做“有自己生活的角色”，内驱力就是下一层关键结构。

它的核心不是回答，而是生成持续性的“自我推动”。

第一版只要把下面这条链路跑通就够了：

**驱动力状态 -> 小意图 -> 行动/推演 -> 反思 -> 状态更新**

这条链路一旦成立，后面再叠加日记、连续记忆、自我叙事、未完成念头，系统就会开始自然长出“自己的节奏”。
