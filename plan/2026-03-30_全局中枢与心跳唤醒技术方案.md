# 全局中枢与心跳唤醒技术方案

**日期：** 2026-03-30  
**目标：** 为 Neo-MoFox 增加一个全局“中枢层”，把所有聊天流的消息、自己发出的消息、记忆整理结果、未完成念头、自我叙事、待办等内容统一汇入事件流，并通过定时唤醒的方式做集中处理。**DFC 的实时聊天流程保持不变**，中枢是并行存在的后台消费者，拥有自己的上下文与状态，但不接管正常回复。

## 0. 前提纠正

这份方案要先明确一个关键前提：

- DFC 还是按原来的流程工作，收到消息就正常回复。
- 中枢不会替代 DFC，也不会打断 DFC 的即时聊天链路。
- 所有事件会同时进入两条路径：一条给 DFC，一条给中枢。
- 中枢的上下文类似 Claude Code 的 `payloads`，是由事件、状态、摘要组成的内部工作集。
- 中枢醒来后处理的是“堆积起来的事件流”，而不是替代聊天本身。

所以，这不是“中枢接管聊天”，而是“实时聊天链路 + 后台心智链路”并行运行。

---

## 1. 背景与目标

当前仓库里已经有多条“局部心智”链路：

- `diary_plugin` 负责连续记忆
- `unfinished_thought_plugin` 负责未完成念头
- `self_narrative_plugin` 负责自我叙事
- `proactive_message_plugin` 负责主动发消息与等待触发
- `memory_passive_trigger` 负责被动浮现记忆

这些能力都能工作，但它们仍然是“分散的”。问题不在于功能不够，而在于缺少一个统一的中心来回答这些问题：

- 我最近在经历什么
- 我最近在想什么
- 我有哪些事还没做完
- 我现在是否适合主动开口
- 哪些信息适合自然地带进聊天

本方案的目标不是把所有模块硬揉成一个大模型，而是建立一个**实时 DFC + 后台中枢**的双轨系统，其中中枢再配合分层状态系统与查询接口：

1. 所有消息先进入事件流，同时也照常进入 DFC 对话流程。
2. 中枢按时间或条件唤醒，批量处理最近事件。
3. 中枢只写内部状态，不直接替代聊天态回复。
4. DFC 可以查询中枢，获取“我最近在干什么、在想什么、记了什么”的结构化报文。

---

## 2. 核心判断

### 2.1 不建议把心智层做成强制 `think` / 强制 `action`

如果把“思考”强制做成聊天内动作，容易出现两个问题：

- 聊天内容被思考过程污染
- 控制流过度依赖一次次显式调用，难以形成连续状态

对“数字生命体”目标来说，更合适的是：

- 思考存在于中枢层
- 聊天只接收摘要和结果
- 需要时由 DFC 主动查询中枢

### 2.2 中枢应是“内部状态引擎”，不是“发言引擎”

中枢负责：

- 消费事件
- 聚合上下文
- 生成内部结论
- 更新记忆、自我叙事、待办、未完成念头
- 产生可分享摘要

中枢不负责：

- 直接接管 DFC 的全部回复
- 每次醒来都对外说话
- 把长篇内心独白直接写进用户聊天

但中枢会维护自己的内部上下文，形式上可以看成一组持续增长的 `payloads`：

- 最近收到了哪些事件
- 自己刚刚在做什么
- 哪些结论已经形成
- 哪些内容还悬而未决
- 哪些信息可以稍后分享给 DFC

### 2.3 DFC 和中枢是“问答关系”

聊天态 DFC 仍然是对外表达层。  
它可以向中枢发问：

- 我最近在做什么
- 我最近在想什么
- 我有没有没处理完的事
- 我现在适不适合主动聊天
- 有没有值得分享的近期状态

中枢返回报文，DFC 再决定是否表达给用户。

## 3. 总体架构

建议新增一个独立的中枢插件或服务，例如：

- `cognitive_core_plugin`
- 或 `drive_core_plugin` 的升级版

整体分为五层：

### 3.1 事件层

把所有输入统一视作事件，但会同时走两条路径：

- 用户消息
- Bot 消息
- 工具调用结果
- 记忆整理结果
- 自我叙事更新结果
- TODO 生成 / 完成 / 失效
- 心跳唤醒结果

路径 A: 进入 DFC 实时链路，按原流程立即回复。  
路径 B: 进入中枢事件堆栈，等待心跳唤醒后统一处理。

事件层采用 append-only 模式，先记账，再处理。

### 3.2 聚合层

按聊天流、按时间窗口、按事件类型聚合：

- 最近 5 分钟发生了什么
- 最近一轮对话发生了什么
- 哪些事件还没被中枢消费
- 哪些事件已经进入稳定状态

### 3.3 决策层

中枢在唤醒时输出结构化决策：

- 是否整理记忆
- 是否更新自我叙事
- 是否新增 / 调整 TODO
- 是否需要探索性行为
- 是否值得对某个聊天流产生主动分享摘要

### 3.4 状态层

中枢维护一组稳定状态：

- `working_memory`
- `event_log`
- `unfinished_thoughts`
- `self_narrative`
- `todo_queue`
- `recent_activity`
- `shareable_summary`

### 3.5 查询层

提供给 DFC 的工具接口：

- `ask_cognitive_core`
- `get_recent_state`
- `get_shareable_summary`
- `list_active_todos`

DFC 不直接读取全量内部状态，只拿结构化结果。

## 4. 数据模型

### 4.1 事件对象

每条事件建议统一成如下结构：

```json
{
  "event_id": "evt_xxx",
  "ts": 1743330000.0,
  "scope": "global",
  "stream_id": "xxx",
  "event_type": "user_message | bot_message | tool_result | memory_update | narrative_update | heartbeat",
  "source": "default_chatter | proactive_message_plugin | diary_plugin",
  "payload": {
    "text": "...",
    "tool_name": "...",
    "meta": {}
  }
}
```

### 4.2 中枢状态对象

建议按“全局 + 流内”两级保存：

- 全局层：近期主动性、探索倾向、系统级任务
- 流内层：某个 `stream_id` 的最近对话、待办、未完成念头、自我叙事摘要

### 4.3 报文对象

中枢返回给 DFC 的结果建议统一为：

```json
{
  "status": {
    "mood": "平静",
    "energy": "中等",
    "social_drive": "低",
    "focus": "整理近期记忆"
  },
  "recent_events": [],
  "internal_updates": [],
  "todos": [],
  "shareable_summary": "",
  "suggestions": []
}
```

报文必须短、结构化、可读，不能把内部推理过程完整暴露给聊天态。

## 5. 心跳与唤醒机制

### 5.1 定时唤醒

中枢通过统一调度器定期醒来，例如：

- 固定间隔心跳：每 5 分钟 / 10 分钟
- 空闲强化唤醒：长时间没有消息时，降低频率但保留检查
- 事件触发唤醒：消息积累到阈值时立即唤醒一次

### 5.2 唤醒前先堆积消息

所有聊天流的消息先进入事件队列，中枢醒来后一次性消费最近窗口。这里的“堆积”只影响中枢自己的消费节奏，不影响 DFC 的即时回复：

- 最近 N 条事件
- 最近 T 分钟事件
- 未消费事件

这样可以保证中枢看到的是“连续片段”，而不是单点噪音。

### 5.3 无消息时也唤醒

即使没有新消息，中枢仍然周期性醒来，检查：

- 有没有该整理的记忆
- 有没有该收束的念头
- 有没有该更新的自我叙事
- 有没有新的 TODO
- 有没有探索冲动
- 是否需要主动分享近期状态

这一步是“生命体感”的关键，不应只等用户驱动。

---

## 6. 与现有模块的关系

### 6.1 `diary_plugin`

负责把经历写入连续记忆，并向中枢提供可压缩素材。  
建议保留其“按聊天流隔离 + 分层压缩 + prompt 注入”的能力，不重复造轮子。中枢可以从它的状态里取最近片段，拼进自己的内部上下文。

### 6.2 `unfinished_thought_plugin`

负责提供“还没想完”的片段。  
中枢可以定期消费这些片段，决定：

- 继续挂着
- 转成 TODO
- 写入自我叙事
- 压成更高层摘要

### 6.3 `self_narrative_plugin`

负责维护“我如何理解自己”。  
中枢可将阶段性结论写回自我叙事，形成稳定连续性。

### 6.4 `proactive_message_plugin`

负责主动开口的外显行为。  
建议保留它作为外显层，改为：

- 先由中枢判断是否“有想说的话”
- 再由 `proactive_message_plugin` 决定是否把摘要自然说出来

### 6.5 `memory_passive_trigger`

负责在聊天态中被动浮现相关记忆。  
这部分仍然适合作为 DFC 侧的注入增强，但不应承担全局中枢职责。

## 7. DFC 查询接口

建议为 DFC 暴露一个轻量工具：

### `ask_cognitive_core`

输入：

- `question`
- `stream_id`
- `scope`（全局 / 当前流 / 最近事件）

输出：

```json
{
  "ok": true,
  "answer": {
    "summary": "...",
    "status": {...},
    "todos": [...],
    "shareable_summary": "..."
  }
}
```

使用场景：

- DFC 想知道自己最近在干什么
- DFC 想知道最近在想什么
- DFC 想问自己还有什么没整理完
- DFC 想拿一段“近期状态”自然融入聊天

这个接口应该短、稳、可重复调用。

---

## 8. 输出策略

### 8.1 不污染聊天

中枢的内部结论不要原样写给用户。  
对外只输出：

- 一句简短状态
- 一条可分享片段
- 一次自然的主动开口

### 8.2 自然融入聊天

主动分享不应像系统播报，而应像：

- “我刚刚整理了一下最近的片段”
- “我这会儿有点在想一件没想完的事”
- “我顺手把最近的东西捋了一下”

### 8.3 保留内部可观察性

日志中应保留：

- 心跳是否触发
- 中枢消费了哪些事件
- 是否更新了记忆 / 自我叙事 / TODO
- 是否产生了可分享摘要

## 9. 实施分期

### 第一阶段：事件流与心跳骨架

目标：

- 建立全局事件总线
- 建立中枢定时唤醒
- 支持把消息和工具结果统一入事件
- 不改动 DFC 原有即时聊天流程

产物：

- 事件模型
- 心跳任务
- 基础状态文件

### 第二阶段：状态聚合

目标：

- 接入 `diary_plugin`
- 接入 `unfinished_thought_plugin`
- 接入 `self_narrative_plugin`

产物：

- `working_memory`
- `todo_queue`
- `recent_activity`
- `shareable_summary`

### 第三阶段：DFC 查询工具

目标：

- 让聊天态能问中枢问题
- 返回结构化报文
- 用于自然表达和主动分享

产物：

- `ask_cognitive_core`
- `get_recent_state`
- `get_shareable_summary`

### 第四阶段：主动表达与自发探索

目标：

- 中枢能决定是否主动找人聊天
- 能决定是否整理 TODO
- 能决定是否主动探索自己

产物：

- 主动分享策略
- 探索任务策略
- 冷却与节奏控制

---

## 10. 风险与约束

### 10.1 风险：中枢变成新的“全能怪物”

解决方式：

- 不直接写回复
- 不直接消费所有细节
- 只维护结构化状态

### 10.2 风险：状态过多导致污染

解决方式：

- 严格分层
- 只有摘要层可注入 DFC
- 原始事件只留给中枢

### 10.3 风险：心跳过频导致骚扰

解决方式：

- 心跳频率可配置
- 空闲期降频
- 输出层必须有冷却

### 10.4 风险：主动性过强

解决方式：

- 主动分享要先过动机阈值
- 每次主动后进入冷却期
- 允许中枢选择“只整理，不发声”

---

## 11. 成功标准

如果这个方案跑通，系统应该能做到：

1. 所有聊天流消息先进入统一事件流。
2. 中枢会被定时唤醒，并批量处理近期事件。
3. 无消息时，中枢也会决定是否要整理、更新、探索或待命。
4. DFC 能向中枢查询“最近在干什么、在想什么、记了什么”。
5. 中枢能返回短、结构化、可自然融入聊天的报文。
6. 对用户可见的聊天内容不会被内部推理污染。

---

## 12. 建议结论

这个方向值得做，而且比单独强化 `think` 更符合“数字生命体”的目标。

最稳的落地路径不是把所有能力塞进一个模型调用里，而是：

- 用事件流记录发生了什么
- DFC 按原有流程即时回复
- 用中枢定时消化发生了什么
- 用状态层保存“现在变成了什么样”
- 用 DFC 查询中枢，再把结果自然说出来

也就是说：

**中枢负责活着，DFC 负责说话。**
