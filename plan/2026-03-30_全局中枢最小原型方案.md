# 全局中枢最小原型方案

**日期：** 2026-03-30  
**目标：** 先做一个最小原型，只验证“并行存在的中枢”这件事本身。这个原型不接收消息、不做决策、不参与聊天、不注入 prompt、不整理记忆，只保留一个独立运行、可被观察、可被后续扩展的中枢壳子。

---

## 1. 目标边界

这个原型只回答一个问题：

> 系统里能不能存在一个和 DFC 并行运行、独立生命周期管理、但暂时什么也不做的中枢？

因此，最小原型必须满足以下约束：

- 不接收用户消息。
- 不接收聊天流消息。
- 不消费事件流。
- 不做任何决策。
- 不触发记忆整理。
- 不触发自我叙事更新。
- 不触发 TODO 生成。
- 不注入 prompt。
- 不提供给 DFC 任何查询能力。
- 不影响现有聊天链路的行为、时序和输出。

也就是说，这一版只验证：

- 中枢能被创建
- 中枢能被启动
- 中枢能被关闭
- 中枢能与主系统并行存活
- 中枢状态可被观察

---

## 2. 为什么先做这个原型

在正式做“心智中枢”之前，先做一个完全空的并行壳子，有三个好处：

1. 可以先验证插件 / 服务 / 调度器 / 生命周期是否适合挂一个独立中枢。
2. 可以先验证并行运行不会影响 DFC 的即时回复。
3. 可以先把“中枢存在”从“中枢做事”里剥离出来，避免一开始就把系统复杂度拉满。

这一步的本质不是功能，而是架构占位。

---

## 3. 原型形态

建议这个最小原型做成一个独立插件，例如：

- `cognitive_core_plugin`
- 或 `drive_core_plugin`

原型只包含三种组件：

1. `Service`
2. `Plugin`
3. 可选的空 `EventHandler` 或空 `Scheduler`

### 3.1 Plugin

职责：

- 注册中枢服务
- 管理配置
- 在加载和卸载时启动 / 停止后台任务

### 3.2 Service

职责：

- 持有一个最小状态对象
- 记录启动时间、最近心跳时间、运行状态
- 提供 `start()`、`stop()`、`health()` 之类的最基础方法

### 3.3 Scheduler / Heartbeat

职责：

- 周期性唤醒
- 只做空转检查
- 写日志，不做任何业务决策

这一层可以先完全不接事件流，只证明后台任务能稳定活着。

---

## 4. 最小状态模型

最小原型只需要一份极简状态：

```json
{
  "running": true,
  "started_at": "2026-03-30T00:00:00+08:00",
  "last_heartbeat_at": "2026-03-30T00:05:00+08:00",
  "heartbeat_count": 12,
  "last_error": null
}
```

可选字段：

- `version`
- `name`
- `uptime_seconds`

这份状态只用于观测，不参与聊天逻辑。

---

## 5. 最小运行流程

### 5.1 启动

插件加载后：

1. 创建中枢服务实例。
2. 标记 `running = true`。
3. 启动一个后台心跳任务。
4. 记录启动时间。

### 5.2 心跳

心跳任务只做三件事：

1. 更新时间戳。
2. 心跳计数加一。
3. 输出一条日志。

禁止在这一阶段做任何额外事情。

### 5.3 关闭

插件卸载或系统退出时：

1. 停止心跳任务。
2. 标记 `running = false`。
3. 记录停止时间。

---

## 6. 明确不做的事

为了确保这个原型足够“干净”，这一版明确不做：

- 不接收聊天事件
- 不接收工具事件
- 不读取连续记忆
- 不读取未完成念头
- 不读取自我叙事
- 不读不写 TODO
- 不生成摘要
- 不向 DFC 暴露查询工具
- 不写入 prompt
- 不做任何主动行为
- 不做任何认知推理

这一步的价值就是“什么都不做，但确实存在”。

---

## 7. 验收标准

最小原型完成后，应满足以下标准：

1. 系统启动时能看到中枢实例被创建。
2. 中枢后台心跳任务持续运行。
3. 中枢不会影响 DFC 的正常聊天。
4. 系统关闭时中枢能被正常停止。
5. 可以通过日志或 health 状态确认它在运行。
6. 全程没有任何消息处理、决策或 prompt 注入行为。

---

## 8. 下一步扩展顺序

最小原型稳定后，再按下面顺序逐步加能力：

1. 先让中枢只“看见事件”，但不做决策。
2. 再让中枢只保存最近事件摘要。
3. 再让中枢只生成内部状态，不外显。
4. 再加 DFC 查询接口。
5. 最后才接入记忆整理、自我叙事、TODO、主动分享。

这个顺序的原则是：

- 先活着
- 再看见
- 再记住
- 再解释
- 最后才行动

---

## 9. 结论

这个最小原型的意义，不是功能，而是验证架构可行性。

如果这一步能稳定存在，后面再加“事件堆积”“定时唤醒”“内部上下文”“查询接口”，每一步都能明确知道是新增能力，还是原有骨架已经支撑不住。

所以最合适的起点不是直接做会思考的中枢，而是先做一个**什么都不做、但确实并行存在**的中枢壳子。
