# DFC 与 Life 合并：我的想法

**日期**: 2026-04-18

我同意这次分析里的大方向，但我不赞成把“合并”理解成立刻把 DFC 整个吞进 life_engine。那样很容易把一个信息桥接问题，误做成一次高风险的运行时重构。

## 我认同的部分

1. 问题不在“桥不够宽”，而在“桥这个中间层本身就太贵”。
2. DFC 之所以拖累 life，不只是 token，更是状态表达被压扁、延迟被叠加、上下文被切断。
3. 继续做 `Life State`、`consult_nucleus`、`tell_dfc` 这类补丁，最多只能缓解，不会根治。
4. 真正值得做的是统一事实来源，而不是继续给两个系统做翻译。

## 我不赞成的部分

1. 不应该一上来就把 `life_engine` 改成一个“万能 Chatter”。
2. 不应该把“统一架构”直接等同于“把所有对话流都收进同一个执行器”。
3. 不应该再加一层更复杂的桥接摘要，尤其是那种“先压缩、再注入、再补偿”的方案。

我的判断是：
- DFC 现在不是“前台做得不够好”，而是“前台这个角色本身开始过时了”。
- 但“过时”不等于“马上删除”。

## 我认为更稳的路径

### 1. 先统一上下文，不先统一运行时

先做的是：
- 一个共享的 context builder
- 一个共享的 memory / state adapter
- 一个统一的摘要规范

目标不是让 DFC 说得更像 life，而是让两者看到同一份事实。

### 2. 再统一执行策略，不急着统一插件边界

心跳和对话可以先共享同一套“决策原则”和“工具约束”，但入口仍然分开。

也就是说：
- 同一套世界模型
- 同一套记忆
- 同一套提示词骨架
- 但仍保留不同的 turn 类型

### 3. 最后才考虑把 DFC 变成 life 的一种 mode

真正值得合并的不是“代码目录”，而是“运行哲学”：
- 心跳是在想
- 对话是在说
- 但两者用的是同一个脑

如果这个阶段能稳定，再去收敛插件边界。

## 我的底线

如果要做合并，我会把底线定成这三条：

1. 不允许双重事实源。
2. 不允许把动态状态反复压缩后再注入 prompt。
3. 不允许为了“看起来统一”而牺牲 turn 的清晰性。

## 一句话结论

我支持“Life 作为统一主体”，但我反对“为了统一而统一”。  
先统一事实和上下文，再统一执行；先减桥，再合体。
